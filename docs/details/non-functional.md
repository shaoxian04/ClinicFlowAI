# Non-Functional Requirements (SAD §3.2)

- **PDPA compliance (Malaysia PDPA 2010)**: encryption at rest + in transit, RBAC, consent, append-only audit log.
- **Performance targets**: GLM p95 ≤ 5s; pre-visit conversational turn < 3s; GLM call timeout 8s; STT timeout 15s. P2 incident if GLM p95 > 10s.
- **Resilience4j** circuit breaker: opens at > 50% error rate, half-open after 30s. Per-doctor rate limiter.
- **Rollout strategy**: Canary 5% → Beta 25% → Majority 50% → General 100% with quantitative gates. A/B test adaptive-rules-on vs. baseline, target ≥ 20% doctor-edit-rate reduction.
- **Priority matrix**: P0 (data leak) → full freeze; P1 (>5% GLM error / dangerous content) → auto-rollback to Golden image < 15min; P2 ≤ 1h; P3 ≤ 24h; P4 next sprint.
- **Four Golden Signals** (latency, traffic, errors, saturation) per service. Instrument with Micrometer + Actuator (Java) and `prometheus_client` (Python).
- **Correlation IDs** (`X-Correlation-ID`) generated at Spring Boot ingress, propagated to the agent service, used in every log line. End-to-end trace via Loki query `{correlation_id="..."}`.
- **Evaluator latency**: `POST /api/visits/{id}/re-evaluate` p95 ≤ 4s end-to-end (Spring → agent → 6 validators → Postgres write). Per-validator soft cap 1.5s; over the cap the validator is marked unavailable and the orchestrator continues. Block both the **approve** and **finalize** Spring routes on `EvaluatorFindingRepository.countUnacknowledgedCritical(visitId) > 0`; the frontend re-evaluates before each call to surface fresh CRITICALs while preserving doctor-in-the-loop.
- **Validator availability is a partial-failure signal, not an outage.** The SSE `evaluator.done` event carries `validators_run` AND `validators_unavailable: [{category, reason}]`. Frontend renders unavailable validators as "DEGRADED" pills; doctor can still review and finalize when only LOW/MEDIUM/HIGH severities are missing — only unacked CRITICALs block. Alert on `validators_unavailable` rate > 5% over 1h (P3); CRITICAL miss alert via canary visits is a P1.
- **Override audit observability**: every per-finding `evaluator_finding_ack` row carries the doctor's reason (truncated to 255 chars) and is queryable from `audit_log` via `resource_type = 'evaluator_finding_ack'`. Use this stream to spot override-reason patterns that should become Hermes-learnable rules — never auto-mine clinical rules from these reasons (Hermes is documentation-style only; see CLAUDE.md invariants).
