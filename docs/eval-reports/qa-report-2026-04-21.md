## Design Assessment

### 1. Visual Consistency: 5/5
All pages feel like one product. Consistent use of:
- Warm off-white background (`bg-[#f0ece4]` approximately)
- Teal/forest green primary color for active tabs, CTAs, headings
- Terracotta/rust accent for patient role badge, warning states
- Serialist/serif typography for headings with italic emphasis
- Clean sans-serif body text

### 2. Color Palette: 4/5
Teal (#1f5c4a approx.) and terracotta (#b85b3a approx.) palette applied coherently. Landing page CTA is terracotta, portal CTAs are forest green, doctor CTAs are forest green. Role badge differentiates Patient (teal) vs Doctor (salmon). Minor inconsistency: "1 error" dev toast is bright red — this is a dev overlay, not shipped.

### 3. Typography: 5/5
Clear hierarchy throughout:
- H1: Large serif (40-60px), often with italic em for emphasis
- H2: Medium serif
- Eyebrows: Small-caps spaced letters
- Body: Readable 14-16px
- Code/IDs: `<code>` monospace

### 4. Whitespace: 3/5
Landing page has excessive vertical spacing between sections (visible as large blank areas at full-page view). Some sections feel disconnected due to very large top padding. Content pages (portal, privacy, doctor) have good proportional spacing.

### 5. Mobile Responsiveness: 4/5
Landing page mobile: content stacks cleanly. Login mobile: left rail stacks above form. Portal visit detail mobile: single-column medications, proper text wrap. Pre-visit intake on mobile: chat interface accessible. No horizontal overflow detected.

### 6. Accessibility Signals: 3/5
- `role="tablist"` and `role="tab"` properly used on language toggle and phase tabs
- `aria-selected` on active tab
- `role="region"` with labels on RedFlagsCard and FollowUpCard
- `aria-label="Patient portal navigation"` on portal nav
- `aria-label="Visit progress"` on ProgressRail
- Missing: visible focus rings (computed outline was empty, though CSS `:focus-visible` may handle this)
- Missing: skip-to-main link
- Missing: `aria-live` on error banners

### 7. Empty/Loading States: 4/5
- Portal: "0 total" count + error banner on visits load failure
- Doctor visit detail post-visit tab: clear "No preview generated yet. Click Generate patient preview..." guidance
- PatientContextPanel: "Context unavailable — HTTP 500." info banner
- Pre-visit intake: "STARTING..." status with disabled inputs
- Visit detail: "Loading your summary…" while fetching

### 8. Overall First Impression: 4/5
The design is distinctive and professional — the warm parchment palette, serif typography, and leaf/nature motifs create a calming, trustworthy aesthetic appropriate for Malaysian clinic context. The language-switching feature (EN/BM) and PDPA consent flow communicate respect for Malaysian legal and cultural context. Not generic.

**Overall design score: 8/10**

## Issues Found

### Important
1. **MedicationCard CSS missing:** `app/portal/components/MedicationCard.tsx` uses CSS classes `.med-card`, `.med-name`, `.med-meta`, `.med-meta-label` but these are only defined within `.meds-list` scope in `globals.css` (line 735). The portal medication cards render drug name + dose label + frequency label as concatenated inline text with no visual hierarchy. Visible on both desktop and mobile.
   - Fix: Add standalone `.med-card`, `.med-name`, `.med-meta`, `.med-meta-label` CSS rules outside `.meds-list` scope.

2. **Raw status enum exposed to doctor:** `app/doctor/visits/[visitId]/page.tsx` line 641 renders `{detail.status}` directly showing "AWAITING_DOCTOR_REVIEW" — raw backend enum. Doctors see technical strings.
   - Fix: Map status enum values to human-readable labels before rendering.

3. **Login fails (backend not running):** The dev server is running but Spring Boot backend (`localhost:8080`) is not. Login always returns HTTP 500. All authenticated data loading fails. The error messages shown are raw HTTP status codes rather than user-friendly messages.
   - Note: This is an environment issue, not a code defect, but the error message quality is poor: "HTTP 500" vs "We couldn't connect. Please try again."

### Minor
4. **Landing page excessive whitespace:** Between hero section and "How it works" section there is ~400px of empty space at 1440px desktop width. This appears to be animation-related padding that looks broken at full-page screenshot.

5. **Post-Visit Preview shows step "05 / PREVIEW"** but ProgressRail only has 4 steps (01-04). The step numbering is inconsistent.

6. **VisitCard shows visitId slice as title:** `Visit {visitId.slice(0, 8)}` is not user-friendly. Should show date or chief complaint as primary identifier.

7. **Error messages are raw HTTP status codes:** "HTTP 500", "! HTTP 500" in error banners throughout (portal, doctor workspace, pre-visit intake) rather than user-friendly messages.

## Verdict: SHIP WITH FIXES

The design language is distinctive, professional, and culturally appropriate for Malaysian clinics. All major UI components (landing, login, privacy, consent, portal, doctor workspace, visit detail) are implemented and render correctly. The bilingual EN/BM feature works with smooth fade transition. Safety invariants and PDPA compliance messaging are thorough.

The primary blocker for a fully functional demo is the Spring Boot backend not running — this prevents login and all data loading. When mocked with valid data, the authenticated pages render excellently.

Two code-level issues need fixing before a real demo:
1. MedicationCard CSS is missing — the drug cards in the patient portal look broken (labels run together)
2. Raw backend enum "AWAITING_DOCTOR_REVIEW" displayed to doctors — needs a label map

The other issues (excessive whitespace on landing, step numbering, raw error messages) are minor polish items.
